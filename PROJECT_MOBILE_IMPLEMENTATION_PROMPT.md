# EduMind 移动端视频分析与推荐学习系统专属总提示词

> 文档用途：作为本项目后续研发、拆解、编码、联调、验收时统一使用的主提示词。  
> 放置位置：仓库根目录。  
> 适用范围：`backend_fastapi/`、`mobile-frontend/`、`ios-app/` 等。  
> 项目目标：在尽量复用现有能力的前提下，为 iOS 等移动端形态提供完整的视频分析、学习辅助与推荐体验。

---

## 一、项目背景与改造目标

当前仓库已经具备以下基础：

1. `frontend/`：桌面端 Vue 前端。
2. `backend_fastapi/`：FastAPI 后端主工程。
3. `mobile-frontend/`：独立移动端 H5/WebView 前端雏形。

本次改造目标不是重做整套系统，而是在尽量复用现有后端与业务能力的前提下，建设一套面向手机端的学习产品闭环，覆盖以下核心模块：

1. 视频上传。
2. 音视频转录。
3. 内容分析。
4. 笔记系统。
5. 智能问答系统。
6. 知识图谱系统。
7. 推荐学习系统。

最终结果要求：

1. iOS 端与移动 H5 能稳定使用。
2. 先有可运行闭环，再逐步增强体验。
3. 先实现移动端学习主链路，再实现图谱和推荐等高级能力。
4. 所有阶段都可以单独验收，不允许只停留在概念方案。

## 二、移动端连调逻辑（必须遵守）

**本项目为移动端应用开发，所有提示词与联调均遵循以下逻辑：**

1. **真机 = 前端**  
   - 在真机（iPhone/iPad）上运行的是「前端」：即 iOS App 内的 WKWebView 所加载的 H5（来自 `mobile-frontend/` 构建产物），以及该 H5 内的全部 UI、请求与状态。  
   - 真机不运行后端代码；真机上看到的页面、点击、表单提交、上传、登录等，均视为前端行为。

2. **功能实现在后端**  
   - 业务逻辑、数据持久化、鉴权、视频处理、笔记与问答等，均在服务端实现，即 `backend_fastapi/`（FastAPI，默认端口 **2004**）。  
   - 前端只负责展示、交互与发起 HTTP 请求，不实现业务规则或直接读写服务器数据库。

3. **前后端通过端口连调**  
   - 前端通过「API 基地址」（如 `http://127.0.0.1:2004` 或真机时 `http://<Mac的局域网IP>:2004`）访问后端，即通过**端口**进行前后端联调。  
   - 本机浏览器调试：`VITE_MOBILE_API_BASE_URL=http://127.0.0.1:2004`。  
   - 真机调试：同一局域网内，`VITE_MOBILE_API_BASE_URL=http://<Mac的局域网IP>:2004`（Mac 上执行 `ipconfig getifaddr en0` 可得 IP，例如 192.168.1.10）；后端 `HOST=0.0.0.0`，且 CORS 放行 `http://<该IP>:5173`。

凡涉及联调、接口约定、环境配置的提示词，均以本条逻辑为准。

## 三、在此逻辑下的视频上传与登录

### 视频上传

- **前端（真机/浏览器）**：用户在 H5 上传页选择本地视频或填写视频链接，点击上传；`mobile-frontend` 将请求发往 `API_BASE_URL`（即上述端口连调地址）。  
- **请求**：`POST {API_BASE_URL}/api/videos/upload`（FormData 字段 `file`）或 `POST {API_BASE_URL}/api/videos/upload-url`（JSON body `{ url }`）。  
- **后端**：在 `backend_fastapi` 端口 2004 上接收请求，保存文件到 `UPLOAD_FOLDER`（默认 `backend_fastapi/uploads/`），并在数据库 `videos` 表插入记录；返回 `id`、`status` 等。  
- **真机连调要点**：`VITE_MOBILE_API_BASE_URL` 设为 `http://<Mac的局域网IP>:2004`（Mac 终端执行 `ipconfig getifaddr en0` 得到 IP，例如 192.168.1.10）；后端 `.env` 中 `HOST=0.0.0.0`，`CORS_ORIGINS` 包含 `http://<该IP>:5173`。

### 登录

- **前端（真机/浏览器）**：用户在登录页输入账号密码，提交；`mobile-frontend` 将请求发往 `API_BASE_URL`。  
- **请求**：`POST {API_BASE_URL}/api/auth/login`，body `{ username, password }`。  
- **后端**：在端口 2004 上校验用户，返回 `token`、`user` 等；前端将 token 存入本地（如 `localStorage`），后续请求在 Header 中携带 `Authorization: Bearer <token>`。  
- **真机连调要点**：与视频上传相同，确保真机可访问后端端口且 CORS 正确。

## 四、技术路线原则

本项目默认采用以下路线推进：

1. 保留 `frontend/` 作为桌面端，不做大拆大改。
2. 以 `mobile-frontend/` 作为移动端主业务工程，承载移动端共享界面与业务逻辑。
3. 以 `ios-app/` 等原生工程作为 iOS 容器，采用 WKWebView 装载 `mobile-frontend/`。
4. 以后端 `backend_fastapi/` 为统一服务入口，逐步补齐移动端专用接口。

这样做的原因：

1. 能最大程度复用当前 Vue 体系与后端接口。
2. 不同 iOS 形态（独立 App、内嵌 WebView 等）可共享绝大多数页面和业务代码。
3. 能先把上传、转录、分析、笔记、问答跑通，再决定是否需要更重的原生重构。
4. 更适合当前阶段的业务验证节奏。

除非后续明确出现以下问题，否则不建议立即切到 Flutter 或全原生：

1. WebView 性能无法满足核心交互。
2. 离线能力要求显著提升。
3. 大量设备能力深度调用不可避免。

1. 现状梳理与 API 冻结。
2. 移动端基础架构。

8. 推荐学习系统。
9. 性能优化、安全治理、打包上线。

核心原则：

1. 先打通主链路，再做增强能力。

## 九、专属代码规范

以下规范必须被视为强约束，而不是建议。

### 1. 架构规范

1. 页面层只负责展示、交互和调度。
2. API 调用必须集中在 `api/`。
3. 业务编排必须集中在 `services/` 或 `stores/`。
4. 类型定义必须集中在 `types/`。
5. 纯工具函数必须放在 `utils/`。
6. 配置必须集中在 `config/`。
7. 后端路由层只做参数接收与响应拼装，不承载重业务逻辑。
8. 后端业务逻辑必须沉淀在 `services/`，耗时任务必须沉淀在 `tasks/`。

### 2. 函数规范

1. **严禁嵌套函数。**
2. **严禁在函数内部再定义业务函数。**
3. **严禁在组件 `setup()`、事件处理函数、回调函数、生命周期函数内部继续声明 helper 函数。**
4. 所有函数必须单一职责。
5. 单个函数超过 50 行时必须评估拆分。
6. 复杂映射、格式化、适配逻辑必须提取到模块顶层函数或独立文件。
7. 异步流程编排必须拆分为多个具名顶层函数，不允许把完整流程堆在一个处理器里。

禁止示例：

1. 在 `setup()` 里面定义 `handleUpload`，再在 `handleUpload` 里面定义 `buildPayload`、`normalizeResponse`、`handleError`。
2. 在 `onMounted()` 内定义轮询函数，再在轮询函数内部定义状态转换函数。
3. 在接口回调里临时写排序函数、过滤函数、映射函数并直接嵌套调用。

正确做法：

1. 把转换逻辑提取到文件顶层函数。
2. 把接口适配提取到 `adapters`、`services` 或 `utils`。
3. 把页面中复用的事件逻辑提取成独立模块函数。

### 3. 组件规范

1. 一个页面只负责一个主场景。
2. 可复用区域必须拆成组件。
3. 组件 Props 和 Emits 必须明确。
4. 组件内部不允许堆大段数据清洗逻辑。
5. 播放器、上传器、问答输入框、图谱卡片应作为独立组件。

### 4. 状态管理规范

1. 用户信息、Token、学习进度、播放状态必须集中管理。
2. 不允许各页面直接操作本地存储。
3. 本地存储读写必须经过统一封装。
4. 轮询状态、上传状态、问答会话状态必须有明确边界。

### 5. API 规范

1. 接口命名统一使用动词加资源名。
2. 请求层统一处理鉴权头、超时、错误码与重试策略。
3. 响应结构必须统一适配后再传入页面。
4. 上传、轮询、问答、播放请求必须有失败兜底。
5. 分页结构、列表结构、详情结构要统一字段风格。

### 6. 命名规范

1. 文件名、组件名、函数名必须语义化。
2. 不允许使用 `temp`、`test1`、`data2`、`handleAll`、`utilFn` 这类弱语义命名。
3. 布尔值命名使用 `is`、`has`、`can` 前缀。
4. 列表获取函数使用 `fetch`，创建使用 `create`，更新使用 `update`，删除使用 `remove`。

### 7. 样式与移动端规范

1. 移动端优先。
2. 必须适配安全区。
3. 必须避免底部导航遮挡内容。
4. 输入场景必须考虑键盘顶起。
5. 列表与详情必须适配小屏滚动。
6. 不允许整页滥用 `100vh` 导致安卓和 iOS 高度异常。

### 8. 可测试性规范

1. 每个阶段至少给出最小验收路径。
2. 每个核心模块至少有手工测试步骤。
3. 后端新增接口必须补充基本测试。
4. 重要状态流转必须可观测、可定位。
