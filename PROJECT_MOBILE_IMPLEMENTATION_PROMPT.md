# EduMind iOS Mobile Implementation Prompt

> 用途：作为本仓库后续研发、联调、编码、验收的统一实现方向文档。
> 当前结论：本项目已收敛为 iOS-only 架构。
> 唯一有效链路：`mobile-frontend/` + `backend_fastapi/` + `ios-app/`。
> 开发设备：MacBook Pro。
> 目标终端：iOS App（`WKWebView` 加载 `mobile-frontend` 构建产物）。

---

## 一、当前项目边界

本项目不再维护以下方向：

1. 桌面网页版产品线
2. 旧 Flask 后端产品线
3. Android 原生产品线
4. 与 iOS 学习链路无关的演示型或兼容型模块

本项目后续实现必须围绕以下三个目录展开：

1. `backend_fastapi/`
2. `mobile-frontend/`
3. `ios-app/`

---

## 二、必须遵守的系统分层

### 1. 前端 = UI 层

`mobile-frontend/` 负责：

1. 页面展示
2. 路由切换
3. 用户输入
4. 上传交互
5. 处理状态展示
6. 播放器界面与学习流界面

前端不负责：

1. 数据库操作
2. 音视频提取
3. 转录
4. 摘要生成
5. QA 推理
6. 业务规则落地

### 2. 后端 = 实际功能层

`backend_fastapi/` 是唯一真实后端，负责：

1. MySQL 数据读写
2. 视频上传保存
3. FFmpeg 音视频处理
4. Whisper 转录
5. 字幕文件与字幕数据写回
6. 摘要生成与标签提取
7. 笔记、问答、推荐等后续能力

所有“真实功能”都必须落在 `backend_fastapi/`，不能写在前端页面里伪装成功能已实现。

### 3. iOS = 容器层

`ios-app/` 负责：

1. `WKWebView` 容器
2. 本地 Web 资源加载
3. 文件选择与 iOS 权限配置
4. 真机运行与验收

---

## 三、前后端联调原则

### 1. 通过端口联调

- 本机开发默认：
  - 前端调试地址：Vite dev server
  - 后端地址：`http://127.0.0.1:<backend_fastapi/.env 的 PORT>`
- 真机调试默认：
  - 后端地址：`http://<Mac LocalHostName>.local:<backend_fastapi/.env 的 PORT>`

### 2. 后端端口是唯一真实能力入口

所有上传、处理、状态查询、字幕查询、视频详情、问答等请求，都必须走 `backend_fastapi` 端口。

### 2.1 真机默认地址必须跟随后端端口

- 修改 `backend_fastapi/.env` 中的 `PORT` 后，执行 `bash ios-app/sync_ios_web_assets.sh`
- 同步脚本必须将 iOS 原生默认后端地址刷新为 `http://<Mac LocalHostName>.local:<PORT>`
- 不允许继续手工在多个页面、多个文档、多个原生配置里散落改端口

### 3. UI-only 只能作为临时占位

如果页面暂时使用 mock 或 UI-only 数据，必须明确标注为占位态；接入真实功能时，必须切回真实后端接口。

---

## 四、数据库约束

1. 数据库必须为 MySQL。
2. 不要随意修改现有数据库表。
3. 不要为了新功能轻易新增表。
4. 优先适配现有表结构。
5. 如果某能力可以写入现有 `videos`、`subtitles`、`notes` 等表，就不要新建平行表。
6. 任何表结构变更都必须是最后手段，并且需要同步更新文档。

---

## 五、视频主链路必须优先打通

必须优先保证以下闭环：

1. iOS 端选择视频或输入视频链接
2. 前端调用 `backend_fastapi` 上传接口
3. 后端保存视频并写入 MySQL
4. 后端执行音频提取、转录与字幕生成
5. 后端把处理状态、字幕路径、字幕数据写回现有表
6. 后端继续生成摘要与标签，并写回现有 `videos` 表字段
7. 前端轮询状态并展示结果
8. 用户在 iOS 容器里查看详情、字幕、摘要与后续学习功能

如果主链路未通，不要优先做边缘功能。

## 五点一、处理设置的实现边界

处理设置可以在前端页面中配置，但只能作为参数下发给后端，不能在前端直接实现“伪摘要”“伪标签”或伪造处理结果。

处理设置至少应覆盖：

1. 识别语言
2. Whisper 模型
3. 摘要风格
4. 是否自动生成摘要
5. 是否自动提取标签

这些设置应至少作用于：

1. 新上传视频
2. 链接导入视频
3. 视频详情页重新处理
4. 失败任务重试

## 五点二、视频问答实现边界

视频问答必须走 `backend_fastapi` 的真实后端链路，不能在 `mobile-frontend/` 中返回 mock 逻辑冒充已完成。

视频问答实现要求：

1. 前端只提交 `video_id`、问题内容和必要的问答参数
2. 后端必须基于现有字幕、摘要、标签等内容做上下文检索
3. 上下文问答只允许接入通义千问或 DeepSeek
4. 不需要 Ollama 离线问答分支
5. 问答结果应写入现有 `questions` 表，不新增平行表
6. 若字幕或摘要不足以回答，后端必须明确返回“无法确认”，而不是编造答案
7. 若前端以 `stream=true` 请求问答，后端必须返回可解析的阶段进度事件，至少覆盖“已接收、检索中、推理/回答中、整理中、已完成”五个阶段，便于 iOS 端展示真实运行进度
8. 视频问答中的 `通义千问` 与 `DeepSeek` 必须作为两个独立聊天空间处理；至少要在前端状态、本地缓存与请求处理层完成隔离，不能继续共享同一段 history；在不修改现有 `questions` 表结构的前提下，不要把共享数据库历史回灌到两个模型空间

## 五点三、用户认证实现边界

用户认证必须继续基于现有 `users` 表实现，不允许新增认证平行表。

实现要求：

1. 注册至少提供邮箱或手机号其中一项
2. 密码必须满足强密码要求，不能只做前端提示
3. 登录只接受邮箱/手机号与密码
4. 用户密码必须以后端安全存储形式写入 `users` 表，不能只存在前端内存
5. 若需要扩展认证统计或标识字段，只能在 `users` 表增列，不新增用户统计表

## 五点四、笔记系统实现边界

笔记系统现在进入正式建设阶段，但必须建立在当前 iOS-only 架构和现有数据模型之上。

实现要求：

1. 笔记系统默认基于现有 `notes`、`note_timestamps`、`videos`、`subtitles` 表与接口，不要轻易新增平行表。
2. 前端只负责笔记列表、详情、编辑、筛选、入口跳转和时间戳展示；关键词提取、相似笔记、导出、时间戳写入等真实逻辑必须继续放在 `backend_fastapi/`。
3. 笔记必须支持普通笔记与视频关联笔记两类主路径；视频关联笔记至少要能带上 `video_id`，并允许挂接一个或多个时间点。
4. 如果在视频详情、播放器、问答等学习流页面新增“记笔记”入口，必须把视频上下文通过接口参数、路由参数或真实后端数据传递，不能只在前端内存里伪造关联。
5. 在不修改现有表结构的前提下优先完成第一版真实可用版本；如确需扩展字段，优先扩展现有 `notes` 或 `note_timestamps`，并同步更新 SQL、测试和文档。
6. 笔记页不能只停留在 UI-only 占位；列表、创建、编辑、删除、时间戳关联、视频筛选等核心链路必须接入真实后端。
7. 如需直接给 AI 编码助手下达任务，优先使用 `docs/NOTE_SYSTEM_IMPLEMENTATION_PROMPT.md` 作为专用提示词。

## 五点五、视频推荐实现边界

视频推荐必须继续走 `backend_fastapi` 的真实后端链路，不能在 `mobile-frontend/` 中拼接假推荐结果冒充已完成。

实现要求：

1. 站内推荐必须优先复用现有 `videos` 表中的 `title`、`summary`、`tags`、`status` 等信息，先把已上传视频的标签提取、归一化和聚类做扎实。
2. 推荐逻辑应优先根据相同标签或高度相近主题组织站内视频，而不是只按时间倒序返回。
3. 外部推荐候选应由后端通过 B 站 / YouTube / 中国大学慕课的 RSS、feed 或公开元数据链路获取；推荐阶段优先只抓轻量元数据，不直接下载完整视频。
4. 外部候选也必须经过后端标签处理，尽量与站内视频使用兼容的标签标准，便于统一排序与相关推荐。
5. 前端只负责展示站内视频与外部候选，并明确区分来源；外部候选应接回现有链接导入链路，而不是伪装成已入库视频。
6. 在第一版推荐增强里，优先不新增外部推荐平行表；如果确实需要缓存或扩展 schema，必须同步更新 SQL、测试和文档。
7. 如需直接给 AI 编码助手下达任务，优先使用 `docs/VIDEO_RECOMMENDATION_IMPLEMENTATION_PROMPT.md` 作为专用提示词。

## 七、开发与运行规范

### 1. Python 必须使用虚拟环境

所有依赖安装、测试、后端启动前，都必须先创建并激活虚拟环境：

```bash
python3 -m venv .venv
. .venv/bin/activate
```

### 2. 后端启动

```bash
. .venv/bin/activate
pip install -r backend_fastapi/requirements.txt
python backend_fastapi/run.py
```

### 3. 移动端启动

```bash
cd mobile-frontend
npm install
npm run dev
```

### 4. iOS 资源同步

```bash
bash ios-app/sync_ios_web_assets.sh
```

## 八、iOS 验收硬规则

任何 `mobile-frontend/` 改动完成后，不能只说“构建成功”。

必须完成：

1. `npm run build:ios`
2. 同步最新 `dist/` 到 `ios-app/EduMindIOS/EduMindIOS/WebAssets/`
3. 在 iOS `WKWebView` 中验证页面显示
4. 验证点击、跳转、列表、详情、上传等交互没有被遮挡或溢出

对页面改动尤其要检查：

1. 文案不超出页面
2. 按钮不撑破布局
3. 长标题、长链接、长文件名可换行或收缩
4. 小屏 iPhone 下不出现横向滚动

## 十、实施顺序

后续所有实现默认按以下优先级推进：

1. iOS 容器可正常加载页面
2. 上传与处理主链路
3. 字幕与详情展示
4. 播放器与学习流
5. 笔记
6. 问答
7. 推荐学习

原则：

1. 先保证真实后端能力可用
2. 先保证 iOS 真机可用
3. 先保证现有 MySQL 表可承载数据
4. 不做与 iOS 主链路无关的分叉实现
