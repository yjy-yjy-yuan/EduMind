# EduMind 智能体编排与能力强化 — 统一提示词

> **用途**：指导本仓库在「视频学习 / 语义搜索 / 问答」方向上的智能体编排、治理、可观测与交付验收。供研发、联调、自动化助手与代码评审对齐目标与边界。
> **产品链**：唯一有效链路为 `mobile-frontend/` + `backend_fastapi/` + `ios-app/`（iOS `WKWebView`）。
> **决策状态（已定）**：智能体编排**目前仅在 `backend_fastapi` 内模块化实现**，未来可拆独立 `agent-service`；业务焦点**先强化现有「视频学习 / 搜索 / 问答」**，不另起无关产品线。

---

## 一、项目主要目标

在 EduMind 现有 **iOS + H5 + FastAPI + MySQL** 工程链上，建设一套**可交付、可演进、可治理**的学习侧智能能力：在**不破坏**「UI 在前端、真实能力在后端、iOS 为容器」的前提下，引入**多角色智能体编排、资源预算、工具治理、可观测与可复利轨迹**，使能力可迭代扩展，而不是堆叠一次性脚本。

---

## 二、当前必须实现的核心能力（架构 + 业务）

### 架构级（与系统七条要求对齐）

1. **端到端学习主路径可用**：素材接入 → 后端处理（转写、摘要等既有能力）→ 内容消费（播放、笔记、搜索、问答）在 **mobile-frontend + WKWebView** 下稳定可用。
2. **智能体编排最小闭环**：至少 **规划（Planner）与验证（Validator/Reviewer）** 在流程上分离（不同 system 提示、不同工具权限、不同最大步数），禁止单一大 prompt 无边界包揽。
3. **工具与外部动作治理**：凡涉及写库、删文件、调外部 API、长时任务等，必须经过**不可被模型提示词覆盖**的治理层（白名单、参数校验、审计记录）。
4. **可观测与恢复基线**：集中式遥测（见下文「可监控」）；任务中断后的**可恢复状态**（任务状态机 + 临时目录与脏数据清理策略）。

### 业务优先级（已定）

1. **搜索增强（建议第一期）**：在现有语义搜索链路上增加「查询理解 / 改写 / 分步检索策略」等编排能力，检索与索引仍走现有服务与 API。验收：可测的召回/相关性改进与可追踪步骤日志。
2. **问答增强**：在 RAG/上下文拼装流程中增加证据选择与校验类步骤；回答与现有 QA 数据结构与引用习惯对齐。验收：固定用例 + 契约测试。
3. **视频学习路径强化**：与当前视频强绑定：时间轴片段、摘要、跳转与问答联动；媒体处理仍使用现有上传/FFmpeg/Whisper 等管线。验收：端到端冒烟 + iOS WebView 按仓库规则验证。

## 三、暂不纳入范围（扩展时预留）

- 多租户与复杂细粒度权限（若当前为单用户/单机构场景）。
- 完整「训练闭环」自动化（数据脱敏、合规、标注流水线）；**先**做好结构化轨迹落盘与导出接口。
- 复杂多智能体市场或插件生态；**先**模块化提示词与可版本化技能。
- 独立 `agent-service`：**仅预留契约与边界**，当前不实施进程拆分。

---

## 四、UI / UX / 产品风格

- **专业、稳重、可信**：信息密度适中、处理状态可见（进行中/失败可重试）、避免炫技动效。
- **验收终端**：交互与路由以 **iOS WKWebView** 为准；桌面浏览器仅作开发调试。

---

## 五、约束：时间、测试、架构与工具链

### 架构

- **FastAPI** 为默认在线执行层；智能体编排放在 `backend_fastapi` 内，**边界清晰**以便未来迁移。
- **禁止**模型或提示词直接操作系统资源；执行只经由治理层与既有 `app/services/` 能力。

### 测试与提交（TDD + 里程碑）

- **严格 TDD**：红 → 绿 → 重构；每个可交付小里程碑：**测试先失败 → 实现 → 全绿 → 单次 commit**。
- **本地验证**：遵循仓库 `AGENTS.md` / `README.md`：日常以 `scripts/validate_backend_smoke.py`、`compileall`、前端构建、iOS 资源同步等为主。
- **CI**：GitHub Actions 等环境可使用 `pytest` 作回归门禁；与「本地默认非 pytest 为主验证」并存，不冲突。

### 其他纪律

- **API 契约变更**须同步相关文档与调用方。
- **MySQL** 为数据库；**不随意改表**；优先适配现有 schema。
- **mobile-frontend** 交互变更：构建并同步 `ios-app/.../WebAssets/`，PR 中声明是否刷新。

## 六、系统层七条要求（目标验收标准）

| 维度 | 要求 | 实现要点 |
|------|------|----------|
| **Effective** | 多角色、流程隔离 | Planner / Validator 等分上下文与工具集，避免混角色。 |
| **Efficient** | Token 预算与缓存感知提示词组装 | 每请求/每步上限、提示词分段、可缓存键（如查询规范化 hash）。 |
| **Safe** | 工具治理管道，模型不可绕过 | 默认拒绝；白名单放行；审计；高危操作按产品需要二次确认或异步策略。 |
| **Robust** | 第二天仍能继续工作 | 脏临时文件清理、中断任务恢复、进程/长任务泄漏策略（统一任务生命周期）。 |
| **Monitorable** | 集中式遥测 | 建立 `services/analytics/`（或与 `app/` 命名风格统一的集中模块），收敛分散 `print`/零散日志；与现有结构化事件（如搜索 JSON 日志）逐步统一 schema。 |
| **Updatable** | 模块化提示词与可版本化技能 | 提示词分段文件或配置版本；运行时「技能」与 Cursor 编辑器侧 `.agents/skills` 解耦。 |
| **Compounding** | 结构化运行轨迹 | 由系统层记录（请求 ID、步骤、工具调用摘要、引用 ID）；支持后续导出到训练/分析闭环，而非单脚本私藏数据。 |

**说明**：上述为**目标清单**；落地时分阶段验收，避免一次性宣称「已全部达成」。

## 七、`backend_fastapi` 内建议物理结构（当前）

以下目录名为建议，实施时可微调，但**职责**应保持一致：

- `app/agents/`（或 `app/agent_runtime/`）：编排入口、会话/任务状态、角色定义与最大步数。
- `app/agents/governance/`：工具白名单、参数校验、审计写入；**唯一合法「执行」路径**。
- `app/agents/prompts/`：分段提示词与版本标识。
- `app/analytics/` 或 `services/analytics/`：**集中遥测服务**（事件模型、出口、与日志级别策略）。

与现有业务连接方式：**薄适配层**调用 `app/services/search/`、`app/routers/qa.py` 及相关视频处理服务，**避免**编排逻辑散落在多个 router。

## 八、未来拆分 `agent-service` 的边界（仅预留，不实施）

- **迁出**：编排运行时、治理、轨迹、提示词版本服务。
- **保留在 `backend_fastapi`**：MySQL、上传、FFmpeg、Whisper、领域业务服务。
- **契约**：任务 ID、用户 ID、资源引用、工具调用记录、版本号；通过 HTTP/gRPC 等明确接口交互。
- **当前约束**：编排代码**避免**深度绑定 ORM 细节，领域修改通过已有 service 函数进入。

---

## 九、优先复用资产（禁止从零重建）

- `backend_fastapi/`：上传、转写、摘要、QA、搜索等现有管线。
- 已有结构化日志（如搜索侧 JSON 事件）：演进为统一遥测事件，而非并行另一套。
- `mobile-frontend/`、`ios-app/` 与 `sync_ios_web_assets.sh` 同步流程。
- `AGENTS.md`、`CHANGELOG.md`、`.github/workflows/backend-ci.yml` 等规范与 CI。

---

## 十、负责人补充规定（在目标允许范围内）

- **单一事实来源**：任务状态与轨迹以服务端持久化为准，前端仅展示。
- **隐私与留存**：轨迹与日志分级；生产环境禁止落密钥与不必要的原始音视频。
- **性能与成本**：除 token 外，对嵌入批次、外部 API 调用等设上限。
